## 2. Decision: sequence 2a → 2b, one branch, eight independently-green batches

**Recommendation: sequence them, 2a first, as ordered batches on this branch —
not two separate waves and not an interleaved mix.** Rationale:

1. **`ruleset_version` must exist before the first balance-affecting change.**
   2b's power drift is the first change since Wave 1 that alters the
   authoritative diff stream. Landing A1 (stamping + golden traces) first means
   the "history never silently rewrites" guarantee is in force *before* the
   change it exists to catch. Landing 2b first would invert that.
2. **2b reuses 2a's authoring machinery.** Degradation drivers are
   content-authored (drift windows, a grid-stress thread), and the low-power
   band checks are deterministic conditions over world state — the same
   `ThreadCondition` evaluation that A2/A3 generalize into a shared
   `engine/conditions.py`. Building 2b first would duplicate that seam.
3. **2a is behavior-preserving; 2b is not.** All of 2a (thread specs moved to
   content, call variants that don't fire on canonical sequences, seed variants
   that are opt-in) keeps the three canonical traces bit-identical — cheap to
   verify continuously. 2b deliberately changes the diff stream and bumps the
   ruleset version once, in one batch (B1), with fixture regeneration. Grouping
   all replay-breaking changes into a single batch keeps the history legible.

They remain *deliverable* independently: if time runs out after A4, 2a alone is
a coherent, shippable wave and 2b stays on the roadmap unchanged.

### Batch A2 — dynamic thread specs move from engine constants to content

Absorbs the Wave-1 trim: `_DYNAMIC_THREAD_SPECS` + the hardcoded opening
triggers in `consequences._threads_for_turn` (`engine/consequences.py:405-518`)
become authored content. **Behavior-preserving: the A1 golden traces must not move.**

**New shared evaluator**
- `engine/conditions.py` (stdlib-only): `condition_holds(cond, variables,
  factions_by_id)` — extracted from `threads._conditions_hold` and extended for
  A3 (see below). `engine/threads.py` delegates to it.

**Engine model**
- `engine/models.py`: new `ThreadSpec` dataclass:
  `id, title, summary, tags, due_in, repeat_every, escalation_effects,
  escalation_note, resolve_tags, resolve_conditions, resolution_note`, plus an
  `open` trigger group: `open_conditions_all: List[ThreadCondition]`,
  `open_conditions_any: List[ThreadCondition]`, `open_advice_tags: List[str]`,
  `open_decision_types: List[str]`.
  Trigger semantics (exactly reproduces current behavior):
  *opens when* all of `open_conditions_all` hold AND (if `open_conditions_any`
  nonempty) at least one holds AND (if `open_advice_tags`/`open_decision_types`
  both empty → true, else the advice's primary tag is in `open_advice_tags` OR
  the decision type is in `open_decision_types`).
  Mapping check against today's five triggers:
  - concealment: tags `[delay]` OR decision `[DELAYED]`, all `[media_pressure ≥ 45]` ✓
  - oversight: all `[state_oversight_risk ≥ 60]` ✓
  - contractor precedent: tags `[contractor]`, all `[contractor_dependency ≥ 70]` ✓
  - school standoff: any `[school_disruption ≥ 55, public_trust ≤ 44]` ✓ (`< 45` ≡ `≤ 44` on ints)
  - trust collapse: all `[public_trust ≤ 39]` ✓
- `Campaign.thread_specs: List[ThreadSpec] = field(default_factory=list)`.
  **Persistence flag:** a pre-2a ACTIVE campaign resumes with empty
  `thread_specs` and stops opening *new* dynamic threads mid-campaign (already-open
  threads keep escalating). Acceptable for local single-player; called out here
  rather than papered over with an engine-constant fallback.

**Content + validation**
- New file `engine/content/scenarios/northbridge_water_failure/thread_specs.json`
  with the five specs verbatim; register `"thread_specs"` in `loader._FILES`.
- `engine/content/schema.py`: `FieldSpec` for `ThreadSpec`; `DECISION_TYPE_VALUES`
  vocabulary read off `DecisionType`.
- `engine/content/validator.py`: `_validate_thread_spec` — reuses the existing
  thread-schedule checks (effects vars known, note required with effects,
  `due_in ≥ 1`, `repeat_every ≥ 0`, resolve tags/conditions valid) plus trigger
  checks (advice tags ∈ `KNOWN_DECISION_TAGS`, decision types ∈ vocabulary,
  conditions valid, at least one trigger criterion present, unique ids).

**Engine rewrite**
- `consequences._threads_for_turn` becomes a generic evaluation of
  `campaign.thread_specs` (opens `OpenThread` with `due_turn = resolving_turn +
  due_in`); `_DYNAMIC_THREAD_SPECS` and the hardcoded `maybe(...)` calls are
  deleted.

**Tests**

  missing escalation note, bad decision type, duplicate id, authored runtime field).
- `test_thread_lifecycle.py`: assert the five specs load and open under the

- Update `test_content_validation.py::test_loader_builds_expected_campaign_shape`

**No API ripple:** `thread_specs` stays engine-internal (Campaign is a SUBSET

### Batch A3 — branchable / faction-gated call variants

**Condition shape (reuses `ThreadCondition`, per the roadmap)**
- `engine/models.py`: `ThreadCondition` gains `faction_id: Optional[str] = None`
  (additive, defaulted → persistence-safe, validator field-spec picks it up
  automatically). When `faction_id` is set, `variable` names a faction numeric
  field from `FACTION_CONDITION_FIELDS = {trust_in_player, influence,
  current_pressure, risk_tolerance}` and is evaluated against that faction;
  when unset, behavior is exactly today's. `engine/conditions.py` implements
  this once — threads get faction-conditioned resolution for free (documented,
  tested, but no authored thread uses it yet).

**Variant model — on Campaign, not on ClientCall (deliberate)**
- New dataclass `CallVariant`: `id: str`, `conditions: List[ThreadCondition]`
  (all must hold), `call: ClientCall` (a complete call body; its `turn` must
  equal the slot's turn; nested variants are structurally impossible because
  variants don't live on `ClientCall`).
- `Campaign.call_variants: Dict[int, List[CallVariant]] = field(default_factory=dict)`
  keyed by turn. Keeping variants off `ClientCall` means **zero ripple** to
  `ClientCallModel`/TS (`ClientCall` is a FULL_ASDICT pair; adding `variants`
  there would force API/TS/fixture churn for a purely engine-side concern).

**Deterministic selection — once per turn, before the `_modulate` seam**
- New `engine/calls.py`: `resolve_call(campaign, turn) -> Optional[ClientCall]`
  — first variant in authored order whose conditions all hold, else the base
  call. Pure; no randomness.
- `Campaign.current_call()` delegates to `resolve_call` — so every existing
  consumer (presentation `_current_model`, `rules._resolve_decider`,
  `rules._modulate`'s profile/red-lines/primary-ids, memo call linkage) sees
  the selected variant with **no duplication of the mediation seam**. The
  variant carries its own `decision_profile`/`primary_advice_ids`, which is
  exactly how "a caller whose trust collapsed opens with a different ask"
  reaches the decision logic.
- **Pinned pitfall:** `advance_turn` must resolve the call **once at entry**
  and pass it down. World state mutates mid-turn, and
  `consequences._immediate_for` currently re-fetches
  `campaign.client_calls.get(resolving_turn)` *after* diffs are applied
  (`engine/consequences.py:243`) — re-evaluating variant conditions
  post-mutation could select a different variant than the one that decided.
  Change `_immediate_for` (and `build_consequence_stack`) to accept the
  resolved call. Same for `campaign_service.submit_advice`'s canon/memo call-id
  lookup at `campaign_service.py:504` — use the recorded resolved id.
- Replayability: between turns state is frozen, so selection at presentation ==
  selection at resolution; a replayed advice sequence re-derives identical
  variants. For the record (design rule: every change explainable):
  `TurnResult.call_variant_id: Optional[str] = None` (additive) — **contract
  ripple**: `TurnResultModel` + TS `TurnResult` + fixture regen (TurnResult is
  a FULL_ASDICT pair).

**Content + validation**
- `calls.json` entries gain optional `"variants": [{id, conditions, call}]`.
  Validator: each variant's `call` is validated with the full existing
  `_validate_call` (same turn, same reference checks, same primary-advice
  rules); conditions validated (faction ids exist; faction-scoped fields ∈
  `FACTION_CONDITION_FIELDS`; ops/thresholds as today); variant ids unique
  across the bundle; ≥1 condition per variant.
- Author 2 variants (new narrative content, no placeholder lore):
  - Turn 4 (Utility Contractor) when `utility_contractor.trust_in_player ≤ 25`:
    the caller opens hostile — different ask, different primary options,
    tighter `off_brief_tolerance`.
  - Turn 6 (State Liaison) when `state_oversight_risk ≥ 60`: the liaison opens
    in intervention posture rather than advisory.
  Neither condition holds on the three canonical sequences (verified by trace:
  oversight peaks at 42 in SURVIVAL, 35 in contractor-spam by turn 6, 48 delay-spam
  turn 6 — the delay-spam turn-6 value is within 12 of the gate, so the variant
  threshold will be checked against the trace before finalizing) → A1 goldens
  and frontend fixtures for turns 1–3 stay bit-identical; if any authored
  threshold would fire on a canonical trace, the threshold moves, not the trace.

**Tests**
- Engine: force contractor trust ≤ 25 → `current_call()` returns the variant;
  the decision explanation carries the variant's mandate (proves selection
  precedes `_modulate`); base call when conditions don't hold; first-match
  order; run-vs-run determinism with a variant active; canonical goldens
  unchanged.
- Content-validation `test_reject_*`: unknown faction in condition, bad faction
  field, variant turn mismatch, duplicate variant id, variant call failing call
  rules.
- Contract: `CallVariant` needs no Pydantic model (never serialized to API);
  `TurnResultModel.call_variant_id` added to schema + TS; fixtures regenerated.

### Batch B2 — degradation bands + capability gating (engine + backend)

**Engine: `engine/degradation.py`** (stdlib-only, pure, computed on demand,
never persisted — same contract as `engine/endings.py`):

| band | power_stability | gates |
| --- | --- | --- |
| NOMINAL | ≥ 55 | none |
| STRAINED | 35–54 | **live feeds lost**: dashboard/evidence values carry stale stamps (`last_live_turn`) |
| DEGRADED | 15–34 | + **memo drafter offline**: `ai_available=false` diegetically; deterministic system drafts only |
| CRITICAL | ≤ 14 | + **subsystem choice required** (B4): only the powered subsystem functions |

- `assess_degradation(campaign) -> DegradationStatus` dataclass: `band`,
  `power`, `live_feeds: bool`, `ai_operational: bool`,
  `requires_power_allocation: bool`, `last_live_turn: int`, `reason: str`.
  `last_live_turn` = the latest resolved turn whose end-of-turn
  `power_stability ≥ 55`, reconstructed by walking `turn_history[*].diffs`
  (old/new values are on the record; turns where power didn't move inherit the
  prior value) — deterministic, no new persisted state.
- Band thresholds live in `degradation.py` as named constants (legible, like
  `FAILURE_THRESHOLDS`).

**Backend seams**
- `_system_status()` (`campaign_service.py:98`) consumes `assess_degradation`:
  `ai_available = settings.ai_live and status.ai_operational`; diegetic
  `model_status` when gated ("Model access offline — grid power below
  sustaining threshold"); `SystemStatusModel` gains `degradation_band: str`,
  `live_feeds: bool`, `last_live_turn: int`, `requires_power_allocation: bool`.
  (`SystemStatusModel` is exempt from the dataclass-parity matrix but IS in
  `TS_INTERFACE_TO_MODEL` — so TS `SystemStatus` must gain the same four
  fields; fixtures regenerate.)
- Memo drafter gating: `draft_memo` / `create_memo(mode="ai")` /
  `submit_advice`'s memo branch consult the band; when DEGRADED or worse they
  skip `run_artifact` and take the existing `fallbacks.memo_fallback` path
  (provenance `deterministic_fallback`, `fallback_used=True`) — the AI
  boundary invariant is untouched, the fallback machinery already exists.
- World-state freshness: when `live_feeds` is false, `_current_model` labels
  `world_state.last_verified` with the stale stamp
  (`"Turn N · LAST VERIFIED — live feed lost (deterministic)"`, N =
  `last_live_turn`). No `Document` model change: docs already carry
  `turn_number` freshness; staleness is presentation derived from
  `system_status` (avoids a `Document` contract ripple).

**Tests:** band boundary unit tests (54/55, 34/35, 14/15);
`last_live_turn` reconstruction; service tests with power forced low —
`ai_available` false even when `ai_live` is monkeypatched true, memo falls

### Batch B4 — the low-power choice (HO#8)

**Engine**
- Vocabulary `PowerAllocation`: `MODEL_ACCESS` / `COMMUNICATIONS` / `LIVE_DATA`.
- `advance_turn(campaign, advice_id, cited_document_ids=None,
  powered_subsystem: Optional[str] = None)`. Deterministic strictness: required
  iff `assess_degradation(campaign).requires_power_allocation` at entry; typed
  errors both ways (`PowerAllocationRequired` / `PowerAllocationUnavailable`).
- Recorded on the turn: `TurnResult.powered_subsystem: Optional[str] = None`
  (additive → `TurnResultModel` + TS + fixtures).
- **First-cut effects are capability gates, not diffs** (deliberate: zero
  balance impact; canonical traces never reach CRITICAL, and the 2-point budget
  margin is not spent on this):
  - `LIVE_DATA` unpowered → citations rejected for that turn (typed error
    `evidence_unverifiable` at the service layer; the engine refuses
    `cited_document_ids` when live data is down) and feeds stay stale.
  - `MODEL_ACCESS` unpowered → drafter hard-forced to system fallback.
  - `COMMUNICATIONS` unpowered → `DecisionExplanation.memory` and
    `caller_disposition` are withheld from the presentation (informational
    loss only; adherence inputs unchanged — the mediation stays deterministic
    and unmodified).
  Numeric costs for unpowered subsystems are **trimmed** to the roadmap (§7).

**Backend — idempotency (the hint, honored)**
- `AdviceSubmissionRequest.powered_subsystem: Optional[str]` (enum-validated).
- `request_fingerprint()` (`campaign_service.py:247`) adds
  `"powered_subsystem": powered_subsystem` to the canonical dict —
  **it joins the fingerprint exactly like `cited_document_ids`** so replaying
  the same key with a different subsystem choice is a
  `idempotency_key_conflict`, and an identical retry replays. Flag: adding a
  key changes fingerprints of *new* requests; a pre-upgrade stored idempotency
  record replayed with an identical payload would now mismatch → KEY_CONFLICT
  instead of replay. Local single-player, in-flight-retry-across-an-upgrade
  only; documented, not mitigated.
- Service validation mirrors the engine rule (required iff CRITICAL) with
  typed 409 errors, checked *before* the engine call inside the transaction.

**Frontend**
- When `system_status.requires_power_allocation`, the advice workbench shows a
  blocking three-way allocation choice (diegetic: "Auxiliary power supports one
  subsystem this cycle"); the choice is included in `submitAdvice` and disables
  the gated affordances (citation picker / AI draft / disposition panel) live.
- Tests: vitest for the chooser + payload; API tests for both typed errors,
  fingerprint conflict on differing choice, replay on identical choice;
  engine determinism test with a CRITICAL-band campaign.
